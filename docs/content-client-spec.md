# 콘텐츠 클라이언트 개발자 기획서

## 목적
콘텐츠 클라이언트는 물리적 디바이스에서 콘텐츠를 실행·관리하며, CMS 서버와 통신해 인증·다운로드·업데이트·텔레메트리를 수행한다.

## 핵심 책임
- 디바이스 등록 및 인증 흐름 구현
- 패키지/콘텐츠 다운로드, 무결성 검증 및 로컬 캐시 관리
- 콘텐츠 실행(런처) 및 버전 검증
- 실행 로그·사용 통계(텔레메트리) 수집 및 전송
- 네트워크 불안정 환경에서의 복원력(재시도, 백오프, 오프라인 모드)

## 실행·인증 플로우 (권장)
1. 기기 부팅 시 `POST /api/devices/register`로 등록 또는 등록 확인
2. 콘텐츠 실행 요청 시 `POST /api/auth/verify`로 인증
3. 인증 성공 시 메타데이터(패키지 버전, 다운로드 URL, 해시)를 받아서 로컬 캐시 확인
4. 로컬에 없거나 버전 불일치면 다운로드 → 해시/서명 검증 → 설치
5. 실행 로그 및 사용 통계는 배치로 전송(네트워크 상태 고려)

## 로컬 저장소 및 캐시 전략
- 저장 경로 분리: `content/`(실행 파일), `meta/`(메타데이터), `logs/`
- 저장소 용량 관리: LRU 기반 또는 패키지 우선순위 정책
- 무결성: SHA-256 해시 검증 + 서명 확인(서명 제공 시)

## 인증/보안
- HTTPS만 사용. 서버에서 제공하는 토큰/세션을 안전하게 저장(운영체제 키스토어 권장).
- One-time 인증 흐름: 최초 인증 후 로컬에 제한적 토큰을 저장해 오프라인 동작 지원(유효기간 명시).

## 네트워크·오프라인 정책
- 재시도: 지수 백오프(예: base 2, max 5회)
- 오프라인 모드: 인증 실패 시 로컬 허용 정책(정책은 서버와 협의 필요)
- 다운로드 중단·재시작: 청크/범위 요청 또는 임시 파일 체크포인트

## 텔레메트리 및 로깅
- 이벤트 유형: 인증, 다운로드 시작/완료/실패, 실행 시작/종료, 오류
- 전송 주기: 비실시간(예: 5분 단위 배치) 또는 네트워크가 유효할 때 즉시
- 로그 포맷: JSON 라인(logstash/ELK 친화)

## 주변 장치 연동
- 터치, 센서, 프로젝터 등과의 통신은 제품별 드라이버/플러그인으로 추상화
- 하드웨어 실패·연결 오류는 별도 이벤트로 서버에 전송

## 테스트·검증
- 시나리오: 등록→인증→다운로드→무결성검사→실행
- 네트워크 조건: 지터/패킷 손실/오프라인 시나리오 테스트
- 회귀: 패키지 버전 업/다운 시 클라이언트 동작 확인

## 배포·업데이트
- 클라이언트 소프트웨어 자체 업데이트는 안전 모드(업데이트 실패 시 롤백) 제공
- 패키지 업데이트는 서버의 버전 필드 기준으로 결정

## 샘플 요청/응답 예시
- 등록 요청
```json
POST /api/devices/register
{
  "deviceId":"dev-001",
  "mac":"AA:BB:CC:DD:EE:FF",
  "productId":"product-xyz"
}
```
- 인증 요청
```json
POST /api/auth/verify
{
  "deviceId":"dev-001",
  "contentId":"content-123"
}
```
